# Research: Simplify Constraint API & Deep Attribute Type Safety

## R1: Recursive Valibot Schema for Nested Attributes

**Decision**: Use `v.lazy()` with a discriminated union to define a recursive `AttributeSchemaNode` valibot schema.

**Rationale**: Valibot already uses `v.lazy()` for `ConditionExpressionSchema` in the codebase (schema.ts:56). The same pattern works for recursive attribute schemas. The discriminated union (`kind: 'primitive' | 'object' | 'array'`) enables exhaustive matching in both validation and codegen.

**Alternatives considered**:
- Flat union without `kind` discriminator: harder to disambiguate at runtime, requires type guards
- JSON Schema subset in YAML: more verbose, doesn't feel native to the existing policy format

## R2: YAML `string[]` Shorthand Normalization

**Decision**: Normalize `string[]`/`number[]`/`boolean[]` shorthand in the YAML parser's pre-processing step, before valibot validation.

**Rationale**: The YAML parser (`parser.ts`) already performs pre-processing (loading raw YAML, then passing to valibot). Adding shorthand expansion here keeps the valibot schema clean — it only validates the canonical `AttributeSchema` form. JSON loader doesn't need shorthand support since JSON naturally can't express `string[]` syntax.

**Alternatives considered**:
- Valibot custom transform: couples syntax sugar to validation schema, makes the schema harder to understand
- Post-validation normalization: risk of mixed representations leaking through

## R3: Nested Object Detection in YAML

**Decision**: In YAML, when an attribute value is a plain object (not matching `type: array` + `items` pattern), treat it as a nested object type. Each key-value pair becomes a field in the nested object's `AttributeSchema`.

**Rationale**: YAML naturally represents nested objects as maps. `address: { city: string, zip: string }` already parses as a JS object `{ city: "string", zip: "string" }`. The pre-processor converts this to `{ kind: 'object', fields: { city: { kind: 'primitive', type: 'string' }, zip: { kind: 'primitive', type: 'string' } } }`.

**Disambiguation rule**: If an object has both `type: "array"` and `items` keys, it's treated as an array declaration. Otherwise, it's a nested object (even if it has a `type` key — `type: string` is just a flat attribute named "type").

## R4: Depth Enforcement Strategy

**Decision**: Enforce max nesting depth of 3 during the pre-processing/normalization step, before valibot validation.

**Rationale**: Catching depth violations early produces better error messages with precise YAML path information. The normalizer already walks the attribute tree recursively, making depth tracking trivial.

## R5: ConstraintResult Type Redesign

**Decision**: Replace the three-way union with `{ ok: true; constraint: Constraint | null } | { ok: false }`.

**Rationale**: The current union requires callers to check for three different property names (`unrestricted`, `forbidden`, `constraints`). The new type uses a single boolean discriminator (`ok`) and collapses unrestricted to `null`. This is a common Result pattern in TypeScript.

**Impact analysis**:
- `constraint-builder.ts`: Change return statements (lines 57, 72, 143, 175, 178, 181)
- `constraint-types.ts`: Replace `ConstraintResult` type definition
- `engine.ts`: Update `buildConstraints` and `translateConstraints` method signatures
- `translator.ts`: Update error messages referencing old sentinels
- `packages/drizzle`, `packages/prisma`: No adapter code changes needed (adapters implement `ConstraintAdapter`, not `ConstraintResult`). Only resolver type annotations may need updating.
- Type tests (`constraint-pipeline.test-d.ts`): Update assertions

## R6: ResourceResolver Return Type

**Decision**: Change `ResourceResolver<S, R>` return type from `Promise<Record<string, unknown>>` to `Promise<Partial<S['resourceAttributeMap'][R]>>`.

**Rationale**: The schema's `resourceAttributeMap` already maps resource types to their attribute shapes (generated by codegen). Using `Partial<>` allows resolvers to return a subset of fields (not all fields may be fetched). When `S = DefaultSchema`, `Partial<Record<string, unknown>>` ≡ `Record<string, unknown>`, preserving backward compatibility.

## R7: Dot-Path Validation in Policy Validator

**Decision**: Add a new `validateAttributePaths()` function to `validator.ts` that walks all condition expressions and checks `$resource.*` and `$actor.*` dot-paths against declared attribute schemas.

**Rationale**: The validator already walks conditions (see `extractActorAttributes` and `validateActorAttributeRefs`). Extending this to validate full dot-paths against the new recursive `AttributeSchema` is a natural extension. The existing validation only checks top-level actor attributes — this expands to nested paths and adds resource attribute validation.

**Rules**:
- `$actor.x.y.z` — validated against actor's `AttributeSchema`
- `$resource.x.y.z` — validated against resource's `AttributeSchema`
- `$env.*` — exempt (no schema to validate against)
- Paths traversing into array types are rejected (e.g., `$resource.members.role` where `members` is `kind: 'array'`)

## R8: Adapter Package Impact (Minimal)

**Decision**: Adapters only need import/type updates, not behavioral changes.

**Rationale**: After analyzing `packages/drizzle/src/index.ts` and `packages/prisma/src/index.ts`:
- Adapters implement `ConstraintAdapter<TQueryMap>` — this interface does NOT change
- Adapters don't reference `ConstraintResult` at all
- The resolver helpers (`createDrizzleResolver`, `createPrismaResolver`) already use `S['resourceAttributeMap'][R]` in their return types
- Only change: update toride peer dependency version to the new major version
- The resolver helpers benefit automatically from the typed return value change in core

## R9: JSON Schema Generation Impact

**Decision**: Update the `generate-schema.mjs` script to handle the new recursive `AttributeSchemaNode` valibot type.

**Rationale**: The `convertSchema` function in `generate-schema.mjs` converts valibot schemas to JSON Schema. It already handles `v.lazy()` for `ConditionExpression`. The same pattern applies to `AttributeSchemaNode`. The generated JSON Schema will include a `$defs/AttributeSchema` entry with the recursive definition, enabling IDE autocompletion for nested attributes.
